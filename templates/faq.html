Getting Started
5 questions
It takes four steps. Upload your model file by dragging it into the Model File zone (supports .pkl, .pt, .pth, .h5, .onnx). Upload test data (.csv, .npy, or .npz). Set your parameters — batch size, number of iterations, and warmup runs. Then click Run Benchmark. Results appear in seconds including latency percentiles, throughput, and memory usage.
Tip: start with 100 iterations and 10 warmup runs. Increase to 500+ iterations for more statistically reliable results.
ModelBench requires Python 3.9 or higher. Install dependencies with:

pip install -r requirements.txt

Then start the server with python app.py and open http://localhost:5000 in your browser. No Docker, no cloud account, no API key needed. The ML framework libraries (torch, tensorflow, onnxruntime) are optional — only install what you need for the models you're benchmarking.
ModelBench supports four major frameworks, auto-detected from the file extension:
scikit-learn (.pkl) PyTorch (.pt / .pth) TensorFlow (.h5) ONNX Runtime (.onnx)

You don't need to specify the framework — ModelBench reads the file extension and loads the model with the appropriate library automatically.
Warmup runs are inference passes executed before the timed benchmark begins. They exist to eliminate cold-start bias — the first few inferences are often slower because of JIT compilation (PyTorch), lazy initialisation, or CPU cache warming.

Without warmup runs, your average and median latency figures will be inflated by these one-time startup costs, giving you a misleading picture of steady-state performance. We recommend at least 10 warmup runs for all models; increase to 50+ for large PyTorch or TensorFlow models.
It depends on what you care about. For quick checks, 100 iterations is sufficient to see if a change made things faster or slower. For reliable p99 latency figures, use 500–1000 iterations — a p99 from only 100 samples represents just the single worst run, which is not statistically meaningful.

For production readiness decisions, aim for 1000+ iterations. Use Async Mode for large iteration counts so the benchmark runs in the background without blocking the UI.

Data & Files
5 questions
Three formats are supported for test data:
.csv — comma-separated, first row may be a header .npy — NumPy binary array .npz — NumPy compressed archive (first array used)

All formats are loaded and converted to a NumPy array before inference. For .csv files, non-numeric columns (like string labels or IDs) are automatically dropped.
The labels file should contain one label per row, in the same order as your test data rows. Supported formats are .csv (single column, no header) and .npy (1D array).

For classifiers, labels should be integers or strings matching the model's trained classes. For regressors, labels should be floats. ModelBench will align label count to sample count automatically if there's a minor mismatch.
Labels are completely optional — skip this upload if you only need latency and performance metrics.
The default upload limit is 100 MB per file in development mode and 50 MB in production. You can change this in config.py by editing MAX_CONTENT_LENGTH.

For very large models or datasets, we recommend loading the model server-side and using a smaller representative subset of your test data (1000–10,000 rows is typically sufficient for reliable latency benchmarking).
Uploaded files are saved temporarily to the /uploads folder on your local machine for the duration of the benchmark, then the result JSON is saved to /results. Your model and data files are never sent anywhere — ModelBench has no outbound network calls and no telemetry of any kind.

If you want to clean up uploads manually, simply delete the contents of the /uploads directory. Result history (JSON files in /results) can be deleted from the History tab in the app.
Yes. Set the Batch Size parameter to match your model's expected input. ModelBench will slice your test data into batches of that size and pass each batch as a single inference call.

Use the Batch Size Sweep feature to automatically compare performance across multiple batch sizes and find the optimal setting for your deployment target.

Understanding Metrics
5 questions
p99 (99th percentile) latency is the latency that 99% of your inference requests complete within. Put differently, only 1 in every 100 requests is slower than your p99 value.

It matters because in production, the slowest requests are what users experience as "the app being slow." A model with 5ms average latency but 200ms p99 means 1 in 100 requests will feel painfully slow. Always check your p99 before deploying — the average alone can hide severe tail latency issues.
Rule of thumb: if your p99 is more than 5–10× your median, you likely have intermittent GC pauses or resource contention worth investigating.
Latency is how long a single inference call takes — measured in milliseconds. It's what matters for real-time applications where a user is waiting for a response.

Throughput is how many predictions your model can process per second — relevant for batch processing jobs, offline pipelines, or high-concurrency services.

They often trade off against each other. A large batch size increases throughput (more predictions per second) but also increases latency per call (you wait for the full batch to complete). The Batch Size Sweep helps you find the right balance for your specific use case.
Memory Peak (KB) is the peak heap memory allocated specifically by inference code, measured using Python's tracemalloc. It isolates the memory cost of running your model from the rest of the process.

RAM RSS (MB) is the total Resident Set Size — the total physical RAM the entire Python process is occupying, measured via psutil. This includes the model weights, Python runtime, loaded libraries, and everything else.

Use Memory Peak to understand the marginal memory cost of inference. Use RAM RSS to estimate how much memory to allocate to a container or VM when deploying.
An accuracy of 1.0 (100%) is valid if your labels file contains the same data the model was trained on. This is actually common when using the provided test_data/ sample files, which were generated to match the example models.

For a realistic accuracy assessment, you should use held-out data that the model has never seen. Split your dataset before training, keep the test set separate, and upload that as your test data when benchmarking.
If you're seeing 1.0 on real unseen data, double-check that your labels aren't accidentally in the same order as your predictions.
R² (coefficient of determination) measures how well your model explains variance in the target compared to a baseline that always predicts the mean. A score of 1.0 is perfect, 0.0 means the model is no better than guessing the mean, and negative values mean the model is actively worse than just predicting the mean.

A negative R² usually means one of: (1) your test data has a very different distribution from training data, (2) you're using the wrong labels file, or (3) the model wasn't trained to predict the target column you're evaluating against.

Privacy & Security
3 questions
Absolutely nothing leaves your machine. ModelBench is a fully local Flask application with no outbound network requests, no analytics, no telemetry, and no external APIs. Everything — model loading, inference, metric computation, and result storage — happens entirely on your hardware.
You can verify this yourself: run ModelBench with your network disconnected. It works identically.
Python's pickle format can execute arbitrary code when a file is loaded. This means a maliciously crafted .pkl file can run any command on your machine the moment it's deserialized — there is no safe way to "preview" a pickle without executing it.

ModelBench shows this warning not to block you, but to inform you. If the .pkl file came from your own training pipeline or a trusted source, it's safe to proceed. Never benchmark a .pkl file you received from an untrusted source or downloaded from an unknown location.
Results are stored as plain JSON files in the /results directory on your local machine. There is no database. Each result includes the metrics, configuration, model type, and timestamp.

By default, results older than 30 days are automatically cleaned up. You can change this with the RESULT_RETENTION_DAYS setting in config.py, or set it to None to disable auto-cleanup entirely. You can also delete individual results at any time from the History tab.

Troubleshooting
4 questions
This error means the required library for that model type isn't installed in your Python environment. The fix is straightforward:

• PyTorch (.pt / .pth): pip install torch
• TensorFlow (.h5): pip install tensorflow
• ONNX (.onnx): pip install onnxruntime

You can check which frameworks are currently available from the System tab in the app. Restart ModelBench after installing any new library.
Accuracy metrics only appear when you upload a labels file in the third upload zone. If you didn't include a labels file, only latency and performance metrics are computed.

If you did upload a labels file and still see no accuracy metrics, check: (1) the labels file is a valid .csv or .npy, (2) the number of labels roughly matches the number of test samples, and (3) label values are numeric (not string) for classification. Any error during accuracy computation is shown in the results under an accuracy error field.
ModelBench has a built-in rate limiter to prevent runaway benchmark loops from overloading the server. By default it allows 10 benchmark requests per minute in development mode.

Simply wait 60 seconds and try again. If you're running automated scripts or CI pipelines that need a higher limit, update RATE_LIMIT_MAX_CALLS and RATE_LIMIT_WINDOW_SECONDS in config.py.
Introspection errors on PyTorch models are common and do not affect benchmark results. Latency, memory, and throughput are measured regardless of whether introspection succeeds.

Introspection can fail if the model was saved with torch.save(model.state_dict()) (weights only, no architecture) rather than torch.save(model) (full model). In this case, ModelBench can benchmark inference but cannot extract parameter counts or layer info because the architecture object isn't present in the file.
